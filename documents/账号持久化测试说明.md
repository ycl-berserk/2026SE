# 账号持久化测试说明

> 适用范围：`demo` 后端 + `miniprogram-1` 学生端小程序  
> 目标：验证“不同注册用户的数据能够持久化到 PostgreSQL，并且互相隔离”  
> 更新时间：2026-05-02

## 1. 本次持久化改造覆盖范围

当前版本已经完成以下持久化能力：

- 学生可以使用“学号 + 密码”注册和登录。
- 注册信息会写入 PostgreSQL，而不是只停留在小程序本地缓存。
- 不同学生账号拥有各自独立的个人资料。
- 个人资料可在“个人中心”修改，修改后会真正写回数据库。
- 请假、通知、党团事务等接口继续按 `userId` 隔离数据。
- 已关闭开发环境固定测试用户自动登录。

当前重点验证的数据表：

- `t_user`
- `t_user_role`
- `student_profile`
- 以及后续联动业务表，例如 `certificate_application`

数据库校验 SQL：

```sql
SELECT id, student_no, real_name, phone, email, status
FROM t_user
WHERE student_no IN ('20260001', '20260002');
```

```sql
SELECT user_id, student_no, grade, major, class_name, auth_type, bio
FROM student_profile
WHERE student_no IN ('20260001', '20260002');
```

```sql
SELECT ur.user_id, r.role_code
FROM t_user_role ur
JOIN t_role r ON r.id = ur.role_id
WHERE ur.user_id IN (
  SELECT id FROM t_user WHERE student_no IN ('20260001', '20260002')
);
```

预期：

- `t_user` 中学号唯一
- `student_profile.auth_type` 为 `student` 或 `cadre`
- `t_user_role` 中应绑定 `student` 角色

## 7. 多用户隔离测试

这是本次最重要的测试。

### 7.1 资料隔离

操作步骤：

1. 登录学生 A，修改个人介绍为 `我是学生A`
2. 退出登录
3. 登录学生 B，修改个人介绍为 `我是学生B`
4. 再分别登录 A、B 查看个人中心

预期结果：

- A 只能看到 `我是学生A`
- B 只能看到 `我是学生B`
- 两人的班级、身份类型、宿舍等也应各自独立

数据库校验 SQL：

```sql
SELECT u.student_no, u.real_name, sp.class_name, sp.auth_type, sp.bio
FROM t_user u
JOIN student_profile sp ON sp.user_id = u.id
WHERE u.student_no IN ('20260001', '20260002')
ORDER BY u.student_no;
```

### 7.2 请假数据隔离

操作步骤：

1. 登录学生 A，去“通知”页中的“发起请假”
2. 提交一条请假申请
3. 退出登录
4. 登录学生 B，进入“我的请假”

预期结果：

- 学生 B 不应看到学生 A 的请假记录
- 再切回学生 A，应能看到自己的请假记录

数据库校验 SQL：

```sql
SELECT u.student_no, ca.id, ca.title, ca.status, ca.submit_time
FROM certificate_application ca
JOIN t_user u ON u.id = ca.user_id
WHERE ca.type_code = 'leave'
ORDER BY ca.submit_time DESC;
```

核对点：

- A 的请假记录只属于 A 的 `user_id`
- B 登录后接口只返回 B 自己的数据

## 9. 接口级别测试建议

如果你们要用 Postman 或 Apifox 测，可以重点测这几个接口：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `GET /api/auth/me`
- `GET /api/student/profile`
- `PUT /api/student/profile`
- `POST /api/leave/me/applications`
- `GET /api/leave/me/applications`

推荐校验点：

- 注册后返回 token
- `GET /api/auth/me` 返回当前账号，不串号
- 修改个人资料后再次查询，值一致
- A 的 token 请求不到 B 的数据

## 10. 常见问题排查

### 10.1 启动后端时报数据库连接失败

检查：

- PostgreSQL 是否启动
- `application-postgres.yml` 的账号密码是否正确
- `ruc_platform` 数据库是否存在

### 10.2 注册时报 Flyway 或字段不存在

说明数据库迁移没有完整执行。检查：

- `flyway_schema_history`
- `t_user` 是否已有 `password_hash`
- `student_profile` 是否已有 `auth_type`

可执行：

```sql
SELECT column_name
FROM information_schema.columns
WHERE table_name = 't_user';
```

```sql
SELECT column_name
FROM information_schema.columns
WHERE table_name = 'student_profile';
```

### 10.3 登录后仍然像旧版一样自动进首页

检查：

- 是否用了旧缓存
- 是否重新编译了小程序
- 是否已使用当前版本的 [SaTokenConfig.java](/C:/Users/26417/Desktop/college_cs/2026SE/demo/src/main/java/com/ruc/platform/config/SaTokenConfig.java)

### 10.4 个人资料保存了但数据库没变化

检查：

- 后端日志里 `PUT /api/student/profile` 是否成功
- 前端是否真的带上了 token
- 是否连接错了数据库实例
