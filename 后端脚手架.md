下面我直接给你一份可以开工的后端项目脚手架设计。目标不是“概念正确”，而是让你能按这个骨架把小程序前端先接起来，同时兼顾人大金仓的适配稳定性。

你这个仓库当前已经明确了两件事：一是项目是“学院服务平台”，二是现阶段已经有小程序前端和一份 API 清单，所以后端最合理的第一目标不是全量实现后台，而是优先支撑小程序现有页面和认证流程。仓库 README 说明([Kingbase Help][1]) 里已经列出了认证、学生信息、知识库、党团事务等模块。([GitHub][2])型

我建议你这一版后端用：

* Spring Boot 3.x
* MyBatis-Plus
* Spring Boot JDBC
* Sa-Token
* Flyway
* KingbaseES JDBC 驱动
* Redis 可选
* MinIO 或本地文件存储，可先本地

这样选的核心原因是：金仓官方文档明确提供了 MyBatis-Plus 的适配说明，并在 FAQ 中说明 MyBatis 的分页场景要按 PostgreSQL 方言配置；同时，金仓 FAQ 还专门提到 `spring-boot-starter-data-jdbc` 会遇到 `jdbcDialect` 识别问题，更推荐使用 `spring-boot-starter-jdbc`。所以这里不建议你走 JPA / Spring Data JDBC 这类自动推断较重的路线。([Kingbase Help][1])：

**第一版后端就走“Spring Boot + MyBatis-Plus + Sa-Token + Flyway + Kingbase JDBC”这条线。**

## 二、项目目标边界

先把范围收紧。

你现在最值得优先打通的是这四块：

1. 微信登录 / 平台登录态
2. 首页聚合数据
3. 知识库
4. 党团进度页

原因不是这些功能“最酷”，而是它们最贴近你现有前端和 API 规划。`API.md` 已经列出 `wx-login`、知识库条目、党团事务流程等接口，这说明这几块本来就是第一阶段核心域。([GitHub][3])“**先把小程序学生端跑通**”来设计的，不以“后台全家桶”为目标。

## 三、整体架构

后端建议采用标准四层：

* Controller：接收请求、参数校验、返回统一结果
* Service：业务编排
* Mapper：数据库访问
* Entity / DTO / VO：数据对象

再加一层公共基础模块：

* common：统一响应、异常、工具类、枚举
* config：跨域、鉴权、MyBatis、Flyway、文件上传等配置

整体上是一个**按业务模块分包**的单体应用。你这个项目规模不适合上微服务。

## 五、模块划分

### 1. auth 模块

负责：

* 微信登录
* token 签发
* 当前用户信息
* 登出
* 角色权限查询

第一版接口：

```text
POST /api/auth/wx-login
GET  /api/auth/me
POST /api/auth/logout
GET  /api/auth/permissions
```

其中真正先要写好的，只有前两个。

#### 登录流程

小程序端：

1. `wx.login()` 获取 `code`
2. 调后端 `/api/auth/wx-login`

后端：

1. 用 `code` 调微信接口换 `openid/session_key`
2. 用 `openid` 查 `wx_user_bind`
3. 若已绑定，查本地用户
4. 若未绑定，返回“需绑定”
5. 生成平台 token
6. 返回 `token + 用户信息 + 角色`

这里要注意，**微信只负责换 openid，不要把微信会话当你的业务登录态**。
你的系统应该以自己签发的 token 为准。

#### 返回示例

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "token": "xxxxx",
    "needBind": false,
    "user": {
      "id": 1001,
      "name": "张三",
      "studentNo": "2024xxxx",
      "roles": ["student"]
    }
  }
}
```

## 七、统一返回格式

你最好一开始就统一响应体。

### `Result<T>`

```java
public class Result<T> {
    private Integer code;
    private String message;
    private T data;

    public static <T> Result<T> ok(T data) { ... }
    public static <T> Result<T> fail(Integer code, String msg) { ... }
}
```

### 分页返回

```java
public class PageResult<T> {
    private Long total;
    private Long pageNum;
    private Long pageSize;
    private List<T> records;
}
```

建议：

* `0` 表示成功
* 业务错误用 `400xx`
* 鉴权错误用 `401xx`
* 权限错误用 `403xx`
* 系统错误用 `500xx`

## 九、鉴权设计

### 为什么用 Sa-Token

对你这个项目来说，Sa-Token 比 Spring Security 更轻，更适合：

* 学生 / 管理员 / 辅导员 多角色
* 前后端分离
* 小程序 token 登录
* 课程项目快速落地

### 鉴权逻辑

登录成功后：

* token 放在响应里
* 小程序存到 `storage`
* 后续请求在 header 里带 `satoken` 或 `Authorization`

角色控制：

* 学生端：`student`
* 管理员：`admin`
* 辅导员/支部负责人：`counselor`

接口上直接做注解拦截，例如：

```java
@SaCheckRole("admin")
@PostMapping("/api/admin/knowledge/articles")
public Result<Void> createArticle(...) { ... }
```

金仓官方 MyBatis-Plus 文档给出的示例里，分页插件的方言直接配置成了 `postgresql`。同时，FAQ 里也明确写了，PageHelper 若不指定 `helperDialect=postgresql`，会报“无法自动获取数据库类型”。([Kingbase Help][1])PlusConfig.java` 可以这样写：

```java
@Configuration
@MapperScan("com.ruc.platform.**.mapper")
public class MybatisPlusConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        PaginationInnerInterceptor pagination = new PaginationInnerInterceptor(DbType.POSTGRE_SQL);
        interceptor.addInnerInterceptor(pagination);
        return interceptor;
    }
}
```

如果你后面改用 PageHelper，也必须显式配 PostgreSQL 方言。这个不是“经验建议”，而是金仓 FAQ 里直接提到的适配要求。yway 迁移策略

你不要让 ORM 帮你自动建表。
你应该用 Flyway 管 schema。

目录：

```text
src/main/resources/db/migration
```

版本建议：

* `V1__init_schema.sql`
* `V2__init_auth.sql`
* `V3__init_student.sql`
* `V4__init_knowledge.sql`
* `V5__init_party.sql`
* `V6__seed_base_data.sql`

这样以后你改字段、加表、补基础数据都可追踪。

## 十三、Mapper 设计建议

简单 CRUD 用 MyBatis-Plus 的 `BaseMapper<T>`。
复杂查询，例如知识库搜索、首页聚合、党团进度联表，建议写 XML。

例如：

```java
public interface KnowledgeArticleMapper extends BaseMapper<KnowledgeArticle> {
    List<KnowledgeArticleListItemVO> selectArticlePage(KnowledgeArticleQueryDTO query);
}
```

配套 XML：

```xml
<select id="selectArticlePage" resultType="com.ruc.platform.knowledge.vo.KnowledgeArticleListItemVO">
    select
        a.id,
        a.title,
        a.summary,
        c.name as categoryName,
        a.publish_time
    from knowledge_article a
    left join knowledge_category c on a.category_id = c.id
    where a.status = 1
    <if test="keyword != null and keyword != ''">
        and (a.title like concat('%', #{keyword}, '%')
        or a.summary like concat('%', #{keyword}, '%'))
    </if>
    <if test="categoryId != null">
        and a.category_id = #{categoryId}
    </if>
    order by a.publish_time desc
</select>
```

## 十六、异常处理

建议全局异常处理必须一开始就配好。

### 业务异常

```java
public class BizException extends RuntimeException {
    private final Integer code;
    public BizException(Integer code, String message) { ... }
}
```

### 全局异常处理器

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BizException.class)
    public Result<Void> handleBizException(BizException e) {
        return Result.fail(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> handleValidException(MethodArgumentNotValidException e) {
        return Result.fail(40000, "参数校验失败");
    }

    @ExceptionHandler(Exception.class)
    public Result<Void> handleException(Exception e) {
        return Result.fail(50000, "系统异常");
    }
}
```

## 十九、你当前最该避免的坑

第一，不要一开始按 `API.md` 把所有后台接口全铺开。
第二，不要用 JPA 去赌金仓兼容细节。
第三，不要把知识库问答做成真正大模型接入再开工。
第四，不要把首页拆成很多碎接口。
第五，不要让数据库建表依赖自动推断。

金仓官方材料已经很明确地告诉你：MyBatis-Plus 是有适配路径的，分页要按 PostgreSQL 方言处理，而 `spring-boot-starter-data-jdbc` 会碰到 `jdbcDialect` 问题。你这类课程型项目最重要的是稳定交付，不是框架炫技。([Kingbase Help][1])你的落地建议

你现在就可以开始建一个后端仓库，第一批提交内容应该是：

```text
feat: init spring boot backend with kingbase + mybatis-plus + satoken
feat: add flyway schema and auth module
feat: add knowledge module basic apis
feat: add party overview api
feat: add home aggregation api
```

这样你 GitHub 的提交历史也会很清楚，看起来像一个正常的软件工程项目，而不是最后一口气堆出来的作业。
